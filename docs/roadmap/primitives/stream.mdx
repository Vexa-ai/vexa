---
title: "Stream"
description: "A per-dispatch redis Stream, relayed to the terminal over websocket. Durable + replayable."
---

The **Stream** is how a running dispatch's live output reaches the user. It is a **per-dispatch redis
Stream** keyed `unit:<dispatch_id>:out`.

- The agent (in its container) `XADD`s each event; a relay (agent-api / gateway) `XREAD`s and pushes over
  a **websocket** to the terminal.
- **Streams, not pub/sub** — so it is **durable + replayable**: a terminal that reconnects mid-turn catches
  up, and a **background routine with no live viewer** still records what it did (open it later, replay).
- Used by [chat](/capabilities/chat) (live reply), [live meetings](/capabilities/live-meetings) (cards),
  and as the audit-grade record of any dispatch. Same mechanism Vexa already uses for
  [meeting transcripts](/capabilities/meetings).

## Implemented

The worker `XADD`s normalized UnitEvents — `message-delta` · `tool-call` · `tool-result` · `commit` ·
`turn-complete` — to `unit:<id>:out`; agent-api's `RedisStreamReader` relays them as **SSE** today (the
gateway exposes the same Stream over `ws.v1`). The Stream is **duplex**: interactive follow-up messages
to a live dispatch ride a symmetric `unit:<id>:in`. Code: `agent_api.worker` (out) +
`agent_api.adapters.RedisStreamReader` (relay).
