**Identity** is not just "the subject" — it is the **chain of custody** that makes every dispatch
attributable and authorized. Full detail in [Identity & trust](/architecture/identity-and-trust).

- **subject** = the principal the agent acts **on behalf of** (data + quota owner, e.g. `u_jane`).
- **launcher** = **what triggered** this dispatch (the user, a schedule entry, or an Integration) — it must
  hold a **signed delegation grant** to act as the subject.
- The identity service **mints a short-lived signed dispatch token** (subject + launcher + scope); the
  [Runtime](/primitives/runtime) **attests the workload** (SPIFFE/SPIRE); every **boundary verifies** the
  token — never the agent.

Adopts the **[kagenti](https://github.com/kagenti/kagenti)** stack: SPIFFE/SPIRE + Keycloak (RFC 8693
token exchange) + an Envoy MCP Gateway.

## Implemented — the per-dispatch token

The dispatcher mints a signed token at dispatch time; the worker only **carries** it; every boundary
re-derives authority from the **verified claims**, so a prompt-injected agent can't exceed its grant:

```text
claims = { sub, lch, ws:[{id, mode}], tools:[name], iat, exp }     # identity.v1 DispatchClaims
  may_mount(id, "rw")  → true only where the token granted rw      # the workspace-store boundary
  may_call(tool)       → true only where the token granted it      # the MCP-gateway boundary
```

Dev tier signs with a shared key (HS256) behind the same `mint`/`verify` interface k8s fills with
**SPIRE + Keycloak**. Code: `core/identity/src/identity_core/dispatch_tokens.py` (+ the
`DispatchClaims` shape in `core/identity/contracts/identity.v1/`).
