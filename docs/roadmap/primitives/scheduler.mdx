title: "Scheduler"

The **Scheduler** is the one mechanism that **dispatches agents**. It is **redis** — a sorted-set engine,
not cron files.

It fires on three kinds of trigger:
- **schedule entry** — a time, one-shot or recurring (a "cron" expression is just a recurring entry);
- **integration event** — an [Integration](/primitives/integration) fired it (e.g. new email), *if* a
  subscription is bound;
- **now** — an interactive message ([chat](/capabilities/chat)).

The set of entries is the **user-manageable schedule** — what [routines](/capabilities/routines) edit. A
dispatch entry carries `(subject, launcher, entry-point)`; on fire it asks the
[Runtime](/primitives/runtime) to spawn.

## Implemented

Every trigger funnels through one fan-in — the agent-api `Dispatcher` — which validates the `unit.v1`
envelope, mints the [identity](/primitives/identity) token, and spawns the container. The **in-process
path is retired**: agents never run in the control plane, so even a chat turn is a real container
[spawn](/architecture/execution). Code: `agent_api.dispatch.Dispatcher` + `agent_api.units` (the one
envelope builder).
