---
title: "Workspace"
description: "A git folder in a bucket, with an id and an access mode. Encrypted. Just files."
---

A **git folder stored in a bucket**, identified by `id`, mounted into a dispatch with an access **mode**
(`ro`/`rw`). A dispatch mounts a **list** — typically `system` (ro) + `company` (ro) + `user` (rw).

- **Just files.** Knowledge, plans, tasks, notes, and the agent's **session** are all files. We **do not
  dictate structure** — not in code, not in a contract. git is the durable state *and* the undo.
- **Encrypted at rest** — per-workspace **envelope encryption**; the data key is wrapped by a master key
  the [Identity](/primitives/identity) layer brokers. Decrypted only inside the sandboxed container; keys
  stay in-VPC for air-gap.
- **Mounted, not cloned.** Running a dispatch mounts the folder at `/workspace`; the agent works in place.
  `ro` is materialized read-only; `rw` syncs back.
- The `mode` is the **source of truth for write access** — see [Governance](/architecture/governance).

## Implemented — chats live here too

Continuity is a file. A chat dispatch saves both the **session id** (`.claude/.session`) and the full
**conversation transcript** in the workspace — the runner's transcript dir is symlinked into
`<workspace>/.claude/projects` — so a freshly-spawned container **resumes the conversation from the
workspace**, no warm container needed. Anything the agent should remember (facts, notes, decisions) it
writes as files here and they commit. Code: `agent_api.worker._link_chat_into_workspace`.

<Note>Verified: "my name is jagujagu" → committed `notes/user.md`; a brand-new container then answered
"what is my name?" with "Your name is jagujagu" — resumed purely from the workspace.</Note>
